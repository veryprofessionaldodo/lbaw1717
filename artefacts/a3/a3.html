A3: User Interface Prototype

This user interfaces prototype (or horizontal prototype) has the following goals:

  1. Help to identify and describe the user requirements, and raise new ones;
  2. Preview and empirically test the user interface of the product to be developed;
  3. Enable quick and multiple iterations on the design of the user interface.

This artefact includes three elements:

  1. Overview of the interface elements and features common to all pages;
  2. Overview of the information system from the viewpoint of the users (sitemap); and
  3. Identification and description of the main interactions with the system (at least two), organized as sequences of screens (storyboards).

The interface's descriptions are presented on the end of the document.

1. Interface and common features

Proyekt is a web application based on HTML5, JavaScript and CSS. The user interface was implemented using the Bootstrap framework.

2. Sitemap

A sitemap is a visual representation of the relationship between the different pages of a website that shows how all the information fits together.

The sitemap gives the project team an idea of how the website is going to be build by helping to clarify the information hierarchy.

3. Storyboards

Storyboards are presented to represent some of the main interactions with the system using a sequence of interfaces and explaining how navigation is done between them. They have numbers, which allow easier explanation after the picture. In most screenshots no footer is displayed, but it is present in the page, with the exception of the profile page.

Story 1 - Visitor Interactions

  1. Visitor arrives in the landing page.
  2. Searches public projects, and the results are displayed.
  3. Clicks on one of them and the main project informations are shown, in this case the members and the project’s description.
  4. Attempts to join project, but fails since registration is missing, so a sign in page appears.
  5. After signing in, the user is redirected to his dashboard, where all his projects are visible. In this case, he awaits confirmation from the coordinators on whether or not he is accepted into the project.
  6. After coordinator confirmation he can now access the project.
  7. Once inside the project, all the sprints, tasks and members are within reach.
  8. The user clicks to check the statistics page.

Story 2 - Logged In Users Interactions

  1. A Logged In User is displaying it’s current information.
  2. The user might edit any information from their profile.
  3. The information has been updated.
  4. The projects’s information is being displayed, and the user wants to add a comment to a task.
  5. The comment was added, and so the user decides to check the forum.
  6. In the forum, the user clicks on one of the threads.

Story 3 - Project Coordinator Interactions

  1. User is in dashboard, and wants to create a new project.
  2. The only necessary information to create a new project is its name.
  3. After submitting that information, the project is now visible in the dashboard, with the user being a coordinator.
  4. When inside the newly created project, no sprints are present.
  5. If the user decides to create a new sprint, a popup will appear, to specify the sprint’s name, deadline, and the user can add some tasks to start.
  6. The user checks the tasks for the sprint.
  7. The user wants to add a new task.
  8. The new task appears on the sprint.
